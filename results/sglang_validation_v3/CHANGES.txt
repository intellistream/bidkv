# SGLang Validation v3 — Change Notes

## 实验日期
2026-04（BidKV SGLang 优化迭代）

## 变更原因（相对于 v2）

v2（旧公式）问题：
1. `tokens_freed = current_tokens` 包含了 prompt KV（SGLang RadixAttention
   不会真正释放 prefix-cached prompt KV），导致分母高估
2. `quality_delta` 最大 1.5× 保护不足，BidKV 倾向选出 recompute 代价高的受害者
3. `_proactive_preempt` 在 `select_victims()` 返回空时仍驱逐"非受害者"

## v3 代码变更

### 1. `bidkv_strategy.py` — 新 select_victims 公式

```
tokens_freed = max(1, num_computed - num_prompt)  # 仅输出部分
recompute_norm = max(0.5, num_prompt_tokens / 256)  # 长 prompt → 高重算代价
late_penalty = completion² × 2                      # 接近完成 → 保护
starvation_penalty = num_preemptions × 0.5          # 防止级联驱逐
quality_delta = max(0.1, recompute_norm + late_penalty + starvation_penalty)
U = tokens_freed / quality_delta
```

fallback（output_tokens ≤ 2 时）: `U = current_tokens / (1 + starvation)`

### 2. `scheduler_hook.py` — proactive_preempt 守卫

```python
if scored_reqs[0][0] >= 10.0:
    return  # 无合格受害者时不驱逐
```

### 3. `sgl_kernel/elementwise.py` — PyTorch fallback（环境修复）

sgl_kernel 0.3.21 因 torch ABI 不匹配无法加载 .so（sm86 vs sm90/sm100）。
         rmsnorm, fused_add_rmsnorm, gemma_*rmsnorm, silu_and_mul, gelu*_and_mul
 `_has_cuda_op()` 检查 + PyTorch 纯实现 fallback。

## v3 实验结果（3-run mean，mixed rate=5.7）

| 策略              | TTFT p50 | TTFT p95 | Tput    | 说明               |
|-------------------|----------|----------|---------|-------------------|
| bidkv             | 144ms    | 4791ms   | 4.13rps | v2: 169ms/4930ms  |
| h2o-style         | 115ms    | 655ms    | 5.45rps | v2: 149ms/4734ms  |
| preempt-evict-sjf | 196ms    | 5691ms   | 3.42rps | v2: 174ms/4823ms（高方差）|

### 观察与分析

1. **bidkv 轻微改善**：TTFT p95 4930ms → 4791ms（-2.8%），符合预期。
   新公式减少了对 long-prompt 请求的"误驱逐"。

2. **h2o-style 异常大幅改善**（4734ms → 655ms）：
   - 响应文件大小骤降（1.2MB → 550KB），暗示输出 token 更短
   - 推测：PyTorch sampling fallback（multinomial）与 CUDA sampling kernel
     产生不同的概率分布，导致 h2o-style 下 EOS token 更早被采样
   - h2o-style 的驱逐行为 + PyTorch 采样交互可能放大了这一效应
   - **结论**：此结果受 sampling fallback 影响，不可直接与 v2 对比

3. **preempt-evict-sjf 高方差**：run0(4943ms/4.16rps) vs run1/2(6170ms/3.05rps)
   - run1/run2 吞吐量下降 >25%，原因不明（可能 GPU thermal throttling 或环境扰动）

### 关键约束

- v3 实验使用 **PyTorch sampling fallback**（sgl_kernel CUDA ops 因 torch 2.6/sm86 ABI 不匹配无法加载）
- 所有策略使用相同的 sampling fallback，策略间相对比较仍然有效
- 绝对数值（尤其吞吐量）可能因 PyTorch multinomial 与原 CUDA kernel 行为差异而与 vLLM 实验不可比
